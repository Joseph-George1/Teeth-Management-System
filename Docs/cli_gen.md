# Project Analysis: Thoutha (Teeth Management System)

## Project Overview
**Thoutha** is an integrated ecosystem designed to bridge the gap between dental students and patients in Egypt. The system facilitates the submission of patient cases, automated triaging using AI, case management for students, and secure communication. It is a multi-layered application consisting of a core Java backend, a modern React frontend, and several specialized Python microservices.

---

## System Architecture

### High-Level Components
1.  **Frontend (Thoutha-Website):** A React-based web application providing interfaces for patients and dental students.
2.  **Core Backend (Backend):** A Spring Boot application managing business logic, data persistence, and security.
3.  **AI Chatbot Service (Ai-chatbot):** A hybrid rule-based and LLM-powered triage system for patient symptom analysis.
4.  **Notification Service (Notifications):** A FastAPI-based service for Firebase Cloud Messaging (FCM).
5.  **OTP Service (OTP):** A FastAPI-based service for WhatsApp-based identity verification.
6.  **Admin Dashboard:** A Flask-based management portal for system administrators.
7.  **CORS Proxy:** A middleware layer to handle cross-origin requests between the frontend and backend.

## Data Flow Description
1.  **Patient Case Submission:** A patient enters symptoms via the **Thoutha-Website**.
2.  **Triage Phase:** The request is sent to the **Ai-chatbot**.
    *   *Step A:* Rule-based matching (`questions.json`) attempts to categorize the case (e.g., "Cosmetic Filling").
    *   *Step B:* If rules fail, the **Gemini-powered AI** initiates a chat to clarify symptoms and determine the category.
3.  **Persistence:** The categorized case is stored in the **Oracle Database** via the **Core Backend**.
4.  **Student Interaction:** Students browse available cases. Upon acceptance, the **Core Backend** updates the database state.
5.  **Notifications:** The **Notification Service** triggers a push notification to the patient's device via **FCM** when their case is accepted.
6.  **Security:** Identity is verified via **WhatsApp OTPs** generated by the **OTP Service**.
7.  **Administration:** The **Admin Dashboard** monitors health checks across all services (AI, OTP, Proxy) and interacts with the backend for data visualization.

## Observed Development Patterns
*   **Micro-Frontend/Service Hybrid:** The project uses a large repository (`Teeth-Management-System`) but maintains distinct service boundaries for specialized tasks (AI, Notifications, OTP).
*   **Proxy-Oriented Architecture:** Heavy reliance on proxy layers (`proxy_server.py`, `admin_dashboard.py` as a relay) to handle CORS and service-to-service communication.
*   **Bilingual Support:** Strong emphasis on Arabic/English localization, particularly in the AI Chatbot and database schema.
*   **Scripted Automation:** Extensive use of Bash scripts (`install.sh`, `3mk-zoz.sh`, `astart`) to manage the complex ecosystem of Java and Python services.
*   **Inferred Security:** JWT-based authentication is standard across the backend, supplemented by an obfuscated admin route (`ADMIN_PREFIX`) and WhatsApp-based OTPs for out-of-band verification.
